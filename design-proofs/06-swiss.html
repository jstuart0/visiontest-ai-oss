04 / INDEX OF CAPABILITIES

What the press does,
set as a table of contents.

Seven operations, one pipeline. Each entry is a function; each function is observable end-to-end.

01 / CATCH
Automated capture across platforms.
Screenshots are taken across Chromium, Firefox, WebKit and on real iOS and Android targets, at a matrix of viewports and pixel densities. One authored step, twelve surfaces.
WEB · IOS · ANDROID
PLAYWRIGHT · APPIUM
02 / COMPARE
Pixel and perception.
Every frame is compared twice: once on raw pixels, and once via a structural/semantic pass using SSIM, LPIPS and DINOv2 embeddings. We prefer the one that disagrees less with human judgement.
SSIM · LPIPS
DINOv2 · PIXEL
03 / AUTHOR
Story-based and goal-based authoring.
Write a Storybook-style fixture, or tell us the goal in plain English — "log the user in and go to the settings page." The planner decomposes intent into concrete, replayable steps.
FIXTURE · INTENT
LLM PLANNER
04 / EXPLORE
Exploratory scan — drop a URL.
Provide a URL and we will crawl the application, produce a map of routes, states and interactive affordances, and recommend coverage gaps. Useful as a first contact with an unfamiliar product.
CRAWL · MAP
COVERAGE
05 / APPROVE
Human-in-the-loop approval.
Every visual change is a proposal. Teams accept or reject, with required reviewers, per-project rules, and an audit trail that survives the pull request.
REVIEW · AUDIT
SLACK · GITHUB
06 / STABILISE
Flake detection and quarantine.
Runs with statistically unstable outcomes are flagged, re-run with an isolation harness, and — if they continue to fail on non-deterministic conditions — quarantined automatically pending a fix.
STATS · ISOLATE
QUARANTINE
07 / REPAIR
Autonomous patches, drafted not merged.
When a regression has a plausible local cause, an LLM drafts the patch and opens a pull request against your repository. You read it. You decide. We keep receipts.
PATCH · PR
HUMAN MERGE
